The Benchmark

FuzzingBrain Bench is a benchmark of 50 real zero-day vulnerabilities across 27 widely-used open-source projects — not synthetic CTF binaries, not historical CVEs from the 2010s. Every entry was discovered by FuzzingBrain, reported upstream, and accompanied by a public reproducer.

Source repo → bugs.json See the Trophies

Why another benchmark?

Most fuzzing benchmarks evaluate known bugs — they tell you whether a fuzzer can rediscover something we've already patched. That's useful, but it doesn't tell you whether a system can find bugs in code that is currently being shipped.

Every bug in FuzzingBrain Bench was unknown to its upstream when we reported it. The benchmark therefore measures something closer to what we actually want from automated vulnerability discovery: finding novel bugs in modern code.

What each entry contains

The single source of truth is bugs.json. Today every entry has:

  • id — stable slug
  • project, title — what and where
  • status — fixed or confirmed
  • bug_class — sanitizer tag (heap-buffer-overflow, oom, null-deref…)
  • report — canonical upstream link

Per-bug directories under bugs/<project>/<id>/ are being filled in incrementally with:

  • README.md — description and reproduction notes
  • poc/ — the input(s) that trigger the bug
  • verify.sh — runs the target with the PoC and checks for the expected sanitizer crash
  • Dockerfile (optional) — pinned reproduction environment
  • commit / version info for the vulnerable build
Status: the catalogue is complete. PoC files and verification scripts are landing on a rolling basis — track progress on GitHub.

How to use it

As a corpus

Read bugs.json directly; every entry has a stable id and a permanent link to the upstream report. Cite individual bugs by ID.

As a fuzzing target

Once a bug's directory contains poc/ and verify.sh, you can reproduce the crash with: 

git clone https://github.com/OwenSanzas/FuzzingBrain-Bench
cd FuzzingBrain-Bench/bugs/<project>/<id>
./verify.sh

verify.sh exits 0 when the expected sanitizer report is observed, non-zero otherwise. This makes it a clean target for evaluating automated triage / patching agents.

As an evaluation harness

To score a fuzzer or vulnerability-discovery agent on FuzzingBrain Bench, point it at the vulnerable build of each project and check whether it produces an input that makes the corresponding verify.sh pass. We will publish baseline numbers in a follow-up post.

Citing

If you use FuzzingBrain Bench in a paper or blog post, please cite it as:

FuzzingBrain Bench — a real-world zero-day benchmark.
https://owensanzas.github.io/FuzzingBrain-Bench/

Contributing

The catalogue should grow as FuzzingBrain finds more bugs. To propose a new entry, open a PR against bugs.json and add a directory under bugs/. Every entry must link to a public upstream report.
