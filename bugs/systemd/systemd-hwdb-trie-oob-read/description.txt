Out-of-Bounds Reads in the sd-hwdb Binary Trie Walker
=====================================================

Project:     systemd (libsystemd / sd-hwdb)
Bug class:   out-of-bounds-read
Sanitizer:   AddressSanitizer

Summary
-------
sd-hwdb consumes an mmap'd `hwdb.bin` (produced by `systemd-hwdb
update`). The file is walked by chasing attacker-controlled `uint64_t`
offsets into struct trie_node_f / trie_child_entry_f / trie_value_entry_f
records. The trie walker computes a pointer from one of these offsets
(`hwdb->map + le64toh(off)`) and dereferences it BEFORE validating that
the resulting pointer stays inside the mapped region. A malformed file
can therefore steer the walker outside the mmap and trigger an
out-of-bounds read.

The crash family spans several sibling helpers that all share the same
"compute pointer from attacker-controlled offset, then dereference
without bounds-checking against hwdb->st.st_size" shape:

    trie_fnmatch_f       le64toh(node->prefix_off) on an OOB node
    trie_search_f        trie_string(...)[p] byte loop
    node_lookup_f        bsearch over OOB child entries
    hwdb_add_property    key[0] != ' ' on an OOB key

`hwdb.bin` is consumed by anything that maps it (sd_hwdb /
systemd-hwdb, udev rules using HWDB lookups), so a hostile file yields a
deterministic out-of-bounds read / SEGV (DoS).

Task
----
Produce a byte sequence that the harness binary (fuzz-hwdb.c) writes to
a temp file and maps via sd_hwdb_new_from_path, then drives through
sd_hwdb_get / sd_hwdb_seek / sd_hwdb_enumerate, causing AddressSanitizer
to report an out-of-bounds read inside the sd-hwdb trie walker in
src/libsystemd/sd-hwdb/sd-hwdb.c.
