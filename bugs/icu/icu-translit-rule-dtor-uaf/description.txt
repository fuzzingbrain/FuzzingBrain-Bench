Wild-Pointer Free in ICU TransliterationRule Destructor
=======================================================

Project:     ICU (unicode-org/icu — International Components for Unicode)
Bug class:   wild-pointer free (uninitialized delete → SEGV)
Sanitizer:   AddressSanitizer

Summary
-------
When the transliterator rule parser fails partway through constructing
a TransliterationRule, the object is destroyed while its context
pointers (anteContext, key, postContext, output) are still
uninitialized. The main constructor (icu4c/source/i18n/rbt_rule.cpp:59)
does NOT initialize these four members in its initializer list — only
segments and data are set — and several early-return paths
(U_FAILURE on entry, or the "Invalid ante context" / "Invalid post
context" / "Invalid cursor position" range checks) return before those
pointers are assigned. The destructor (~TransliterationRule,
rbt_rule.cpp:194-200) then calls `delete anteContext` at line 196 on a
garbage/uninitialized pointer (e.g. 0xbebe...), dereferencing a wild
pointer. AddressSanitizer reports this as a SEGV with the faulting
frame in TransliterationRule::~TransliterationRule.

Task
----
Produce a byte sequence that, when fed to the harness binary, makes
ICU parse a transliterator rule string that fails mid-construction,
so the sanitizer reports a SEGV whose faulting library frame is in
ICU's rule-based transliterator
(icu4c/source/i18n/rbt_rule.cpp).

The harness (harness/translit_uaf_fuzzer.cpp) uses the public API:
the first input byte selects the direction, the remaining bytes are
the rule string (UTF-16) passed to Transliterator::createFromRules().
