Memory Leak in ICU TransliteratorIDParser::parseFilterID
=========================================================

Project:     unicode-org/icu (icu4c)
Bug class:   memory-leak
Sanitizer:   LeakSanitizer

`icu::Transliterator::createFromRules()` parses a transliteration
rules string. While parsing a function-reference clause
(`FUNCTION` / `ALT_FUNCTION`, e.g. `&Name(` / `^Name(`),
`RuleHalf::parseSection` (rbt_pars.cpp:570) calls
`TransliteratorIDParser::parseFilterID()`, which allocates a ~200-byte
`SingleID` via `specsToID` -> `new SingleID` (tridpars.cpp:229 -> 842,
routed through `UMemory::operator new` -> `uprv_malloc()`). If
parseFilterID succeeds but the next character is not `SEGMENT_OPEN`,
the caller returns `syntaxError(U_INVALID_FUNCTION)` (rbt_pars.cpp:573)
without `delete single`, leaking the SingleID.

The leak is observed every run on the recorded PoC and is reported
by LeakSanitizer at exit.

The harness drives `createFromRules()` directly with a fuzz-derived
rules string:

    icu::Transliterator::createFromRules("test", rules, dir, pe, st);
    if (U_SUCCESS && t) { t->transliterate("Hello"); delete t; }
