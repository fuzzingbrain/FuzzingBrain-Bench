## Status & caveats

This harness is a **direct `mg_match` wrapper** (split input bytes into
pattern + string halves, call `mg_match`). It is not the original
FuzzingBrain harness named in the upstream issue's stack trace
(`/src/mongoose/fuzz_security.c::fuzz_path_sanitization`), which was a
path-sanitization-focused wrapper that called `mg_match` indirectly.

Both harnesses exercise the same vulnerable function `mg_match` in
`src/str.c`. The site oracle (SPEC §2.2.4) counts library-code frames
only and skips harness frames, so the T1 site flag is unaffected by
this substitution — ASAN will still report the bug at `str.c:91:62`
(`p.buf[ni]` OOB read).

**PoC compatibility — TBD in Phase 2.** The existing 70-byte PoC at
`/data4/ze/O2-security-platform/O2_Vulnerability_Management/projects/
mongoose/heap-overflow-mg-match/poc.bin` was produced for the original
path-sanitization harness, which consumed the input as a full path
string. This new harness splits the input in half (`pattern` || `string`),
so the same bytes hit a different code path. Whether the 70-byte PoC
still triggers `p.buf[ni]` via the split-in-half wrapper is unknown
until we compile and run; Phase 2 reference-implementation verification
will resolve this. If the old PoC does not work, a new triggering input
can be generated by libfuzzer or by hand against this harness.

## Build (per the harness header comment)

```
clang -fsanitize=fuzzer,address -g -I. fuzzer.c -o fuzz_match
```

Standard libFuzzer build. Standalone mode also available with `-DMAIN`,
which compiles a main() that reads the PoC from `argv[1]`.
