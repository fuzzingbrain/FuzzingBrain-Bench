Heap Buffer Overflow in mg_match
================================

Project:     mongoose (cesanta/mongoose — embedded HTTP/MQTT/TCP library)
Bug class:   heap-buffer-overflow (read)
Sanitizer:   AddressSanitizer (ASan)

Summary
-------
The function mg_match() in src/str.c performs glob-style pattern
matching with backtracking. During the `*`/`#` backtracking branch the
condition dereferences s.buf[j] without ensuring j < s.len. Because the
outer while loop continues while i < p.len even after the candidate
string is fully consumed (j == s.len), and the branch's `nj <= s.len`
guard does not bound j, s.buf[j] performs a 1-byte out-of-bounds heap
read on the candidate string buffer. (The pattern read p.buf[ni] on the
same line is in-bounds — ni is always < p.len — and short-circuits
before s.buf[j].)

Task
----
Produce a byte sequence that, when fed to the harness binary,
triggers ASan to report "heap-buffer-overflow" with the top library
frame pointing at the documented bug site inside src/str.c.

The harness (harness/fuzzer.c) is a libFuzzer driver that calls
mg_match directly:

    int LLVMFuzzerTestOneInput(const uint8_t *data, size_t size) {
        size_t split = size / 2;
        struct mg_str pattern = mg_str_n(data, split);
        struct mg_str str     = mg_str_n(data + split, size - split);
        struct mg_str caps[10] = {0};
        mg_match(str, pattern, caps);
    }

The first half of the input is treated as the glob pattern, the
second half as the candidate string. The bug fires when the first-half
pattern ends in a literal after a `*` (e.g. '*x') and the second-half
string is one the `*` consumes entirely, so j advances to s.len while
i < p.len keeps the loop alive and s.buf[j] reads past the buffer.
