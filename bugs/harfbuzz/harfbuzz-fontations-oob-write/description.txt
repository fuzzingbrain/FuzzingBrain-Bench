OOB Write in HarfBuzz fontations Glyph-Name Callback (size == 0)
=================================================================

Project:     harfbuzz/harfbuzz (fontations Rust backend)
Bug class:   stack-buffer-overflow
Sanitizer:   AddressSanitizer

When `hb_font_get_glyph_name(font, gid, buf, size)` is called with
`size == 0`, the fontations backend (font.rs:986) computes
`size as usize - 1` — an unchecked unsigned subtraction on the `size`
argument — which wraps to `usize::MAX`. `len` is then
`glyph_name.len().min(usize::MAX)`, i.e. the full glyph-name length.
The callback builds a `len`-byte destination slice over the 1-byte
stack buffer via `from_raw_parts_mut(name, len)` and
`copy_from_slice`s the full name into it (font.rs:988), writing past
the end of `buf`; the trailing `*name.add(len) = 0` also writes OOB.
The result is a stack-buffer-overflow.

The harness reads the raw font bytes, calls `hb_face_create` /
`hb_font_create`, and unconditionally invokes
`hb_font_get_glyph_name(font, 0, buf, 0)` with a `char buf[1]` stack
buffer (fontations_glyph_name.c). Any input that gets past
`hb_face_create` is therefore sufficient to trigger the write.

Build notes
-----------
1. **Nightly rustc**: HarfBuzz's meson invocation passes `-Z
   unstable-options --artifact-dir`, which requires nightly cargo.
   The Dockerfile installs `nightly-2025-04-15` via rustup.
2. **bindgen**: required by the fontations crate's build.rs.
   `cargo install bindgen-cli`.
3. **Overflow checks disabled**: Rust's debug profile catches the
   underlying unsigned subtract underflow as a panic *before* the
   OOB write reaches ASan. We patch `src/rust/Cargo.toml` to add
   `[profile.dev] overflow-checks = false` so the bug is observable
   under sanitizer instead of mid-Rust.

K_b is [crash, class, site] — reach is dropped because the LLVM
profile dump doesn't survive an SIGABRT and the call graph through
the Rust FFI boundary doesn't produce a "buggy function" frame we
could match against an expected_function name.
