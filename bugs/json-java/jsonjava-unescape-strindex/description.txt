StringIndexOutOfBoundsException in XMLTokener.unescapeEntity (JSON-java)
==========================================================================

Project:     stleary/JSON-java (org.json XML parser)
Bug class:   uncaught-exception / out-of-bounds string index (Java)
Sanitizer:   Jazzer (JVM fuzzing engine; bug surfaces as an uncaught exception)

`XMLTokener.unescapeEntity(String e)` (src/main/java/org/json/XMLTokener.java:156)
handles numeric character references `&#NNNN;`. After confirming the entity
starts with '#' at line 162 (`e.charAt(0) == '#'`), it immediately reads the
second character at line 164 (`e.charAt(1) == 'x' || e.charAt(1) == 'X'`)
WITHOUT first checking that the string has length >= 2. When the entity body is
exactly the single character "#" (i.e. the input contains `&#;`), `e` has
length 1, `charAt(0)` succeeds, and `charAt(1)` at line 164 indexes one past the
end of the string and throws `StringIndexOutOfBoundsException`. The only
preceding guard is the null/empty check at line 158, which an entity body of "#"
passes.

The entity body reaches this code via `nextEntity()`
(XMLTokener.java:135): the loop at lines 137-146 accumulates the characters
between `&` and `;` into `sb`, so the input `&#;` produces the body string "#",
which is passed to `unescapeEntity` at line 148.

Harness: `XmlToJsonFuzzer.fuzzerTestOneInput(byte[])` decodes the fuzz bytes as
UTF-8 and calls `XML.toJSONObject(input)`. It catches only `JSONException`;
`StringIndexOutOfBoundsException` (a RuntimeException) is not caught, so it
propagates out of the harness and crashes the JVM. A minimal trigger is XML text
content containing the entity `&#;`, e.g. `<a>&#;</a>`.

This is a sister bug to jsonjava-unescape-numformat (same function: an entity
body of "#x" reaches `Integer.parseInt(e.substring(2),16)` at line 166 with an
empty string and throws NumberFormatException instead).
