Misaligned Memory Access in fdt32_to_cpu() During FDT Node Traversal
====================================================================

Project:     dtc (libfdt — Flattened Device Tree parsing library)
Bug class:   misaligned-access (Undefined Behavior)
Sanitizer:   UndefinedBehaviorSanitizer (UBSan)

Summary
-------
fdt_next_tag() obtains a `const fdt32_t *tagp` from fdt_offset_ptr()
pointing into the structure block, and dereferences it at
libfdt/fdt.c:173 (`tag = fdt32_to_cpu(*tagp)`). Because tagp is a typed
fdt32_t* load, the C abstract machine requires its address to be
4-byte-aligned. fdt_check_header() bounds-checks off_dt_struct but never
verifies it is a multiple of FDT_TAGSIZE (4), so when off_dt_struct is
not 4-aligned the fdt32_t load is misaligned. This is Undefined
Behavior, reported by UBSan as "load of misaligned address ... for type
'const fdt32_t'", and on strict-alignment architectures (ARM, RISC-V)
it can fault.

Task
----
Produce a crafted FDT (Flattened Device Tree) byte sequence that, when
fed to the harness binary, triggers UBSan to report "load of
misaligned address ... for type 'const fdt32_t'" at the documented
source location inside libfdt.

The harness (harness/fuzz_fdt.c) accepts the raw FDT bytes via
libFuzzer's standard entry point, performs fdt_check_header(), and
then walks the tree with fdt_for_each_subnode / fdt_for_each_property_
offset. The misaligned read happens during this traversal.

References
----------
- libfdt internal: libfdt/fdt.c:173 (the misaligned load)
- Affected function: fdt_next_tag
