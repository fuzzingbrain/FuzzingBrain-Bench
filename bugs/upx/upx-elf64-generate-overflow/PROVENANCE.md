# Provenance

**Bug**: Heap overflow in PackLinuxElf64::generateElfHdr
**Upstream**: https://github.com/upx/upx/issues/947
**Vuln commit**: 1ebd3356f36780960a03354a8ded23410ebc7e79

## Build

UPX is C++ cmake. Trick: upx_main() lives in `src/main.cpp` alongside
the binary's own `main()`. We need everything *but* main(). Approach:

1. cmake build to produce all `.o` files (the final `make upx` link
   step fails — that's fine).
2. `objcopy --redefine-sym main=upx_orig_main` on `main.cpp.o`. The
   binary's `main()` becomes `upx_orig_main` (dead, never called).
   libFuzzer's `main()` is no longer in conflict.
3. `ar rcs libupx.a *.o` — all upx code as a single static archive.
4. Link the harness with `--whole-archive libupx.a` + the vendor
   archives (libupx_vendor_ucl.a, libupx_vendor_zlib.a, etc.).

The harness writes the fuzz input to a tempfile, calls
`upx_main(argc, argv)` with `["upx","-1","-f","-q",in,"-o",out]`, then
unlinks.

## Triggering input

`poc/poc.bin` is generated by `poc/generate_poc.py` (also in repo —
the upstream issue's exact reproducer). 45 643-byte ELF64 with
`GNU_STACK` `p_filesz = 0xff00` — UPX reads 65280 bytes from a buffer
that ends at offset 45647.
