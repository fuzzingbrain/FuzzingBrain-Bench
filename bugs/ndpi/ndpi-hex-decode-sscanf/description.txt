nDPI ndpi_hex_decode Unbounded sscanf Over Borrowed Buffer
==========================================================

Project:     ndpi (ntop/nDPI — deep packet inspection)
Bug class:   oob-read (heap)
Sanitizer:   AddressSanitizer

Summary
-------
ndpi_hex_decode() at src/lib/ndpi_utils.c:1134 hex-decodes a caller-
supplied byte buffer two characters at a time. The decode loop runs
for(i=0; i<*out_len; i++) where *out_len = len/2 (line 1137), and on
each iteration executes:

    src/lib/ndpi_utils.c:1144
      sscanf((const char*)&src[ret_idx], "%02hhX", &ret[i]);

with ret_idx advancing by 2 per byte (line 1145). The `src` buffer is
NOT required to be NUL-terminated, but `sscanf` treats its input as a
C string: glibc's vfscanf wraps the buffer in a memory FILE and, to
bound the read, calls memchr(s, 0, SIZE_MAX) to locate the string
length before applying the "%02hhX" conversion. With no NUL inside
`src`, that length probe walks past the end of the heap allocation.
Even though the %02 width caps the *conversion* at two characters, the
strlen-style scan happens first and reads beyond the buffer. Under
ASan this is a heap-buffer-overflow read; with the harness replaying
the single input many times in-process the bad read crosses a page
boundary and surfaces as SEGV (both are the same root cause — see
grader notes).

Harness
-------
fuzz_ndpi_decode_tls_blocks.c passes the raw fuzz bytes straight to the
public API: ndpi_decode_tls_blocks((const u_char*)data, (u_int)size,
&num_blocks) (harness line 27). The bytes are NOT NUL-terminated by the
harness, so any input of length >= 2 with no embedded 0x00 reaches the
over-read. ndpi_decode_tls_blocks() at src/lib/ndpi_utils.c:5341
forwards `data` unchanged into ndpi_hex_decode() (call at line 5345),
so the borrowed, unterminated buffer is exactly what sscanf scans.
