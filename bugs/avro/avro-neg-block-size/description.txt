Uncontrolled Allocation Size in Avro C File Reader
==================================================

Project:     apache/avro (lang/c)
Bug class:   allocation-size-too-big
Sanitizer:   AddressSanitizer

The Avro C data-file reader reads a per-block size field from an Avro
object-container file using zigzag-encoded varints, which can represent
negative numbers. The decoded size is used as an allocation size without
validating that it is non-negative. A crafted file whose block-size field
decodes to a negative value (e.g. -1) turns into an enormous unsigned
allocation request (~18 EB), which AddressSanitizer rejects as
allocation-size-too-big.

Call chain on a malformed Avro container file:

    avro_file_reader_fp        lang/c/src/datafile.c:529
    └─ file_read_block_count   lang/c/src/datafile.c (block-size alloc)
       └─ allocator            <- requested allocation size 0xffffffffffffffff
