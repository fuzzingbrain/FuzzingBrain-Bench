Negative String Length OOB in Avro C Binary Value Reader
========================================================

Project:     apache/avro (lang/c)
Bug class:   memory-exhaustion (allocation-size-too-big)
Sanitizer:   AddressSanitizer

The Avro C binary value reader trusts an unvalidated, signed varint
string length. The harness picks one of several predefined valid
schemas using the first input byte, builds a generic value for it, then
calls `avro_value_read()` on the remaining bytes. For a string or
map-key field, `read_string()` (lang/c/src/encoding_binary.c:172) decodes
a signed varint length without bounds-checking it, computes
`*len = str_len + 1`, and passes that to `avro_malloc(*len)`. A negative
varint length wraps to a near-SIZE_MAX request (e.g. 0xffffffffffffffe9),
which AddressSanitizer rejects with "requested allocation size ...
exceeds maximum supported size". `read_string()` is inlined into
`read_map_value`, so the closest library frame the sanitizer reports is
lang/c/src/value-read.c:103.

Call chain:

    avro_value_read        lang/c/src/value-read.c:391
    └─ read_value          lang/c/src/value-read.c:368
       └─ read_map_value   lang/c/src/value-read.c:103  (read_string inlined)
          └─ avro_malloc -> avro_default_allocator  lang/c/src/allocation.c:36  ← ASan fires
